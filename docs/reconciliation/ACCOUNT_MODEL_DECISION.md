# Account Model Decision

> Generated by Agent 0 on 2026-07-13.
> This document records the current state of the account system and the required corrections per product decision.

## 1. Product Decision (Frozen)

The project has TWO separate account systems:

### A. Frontend User (普通用户)
- Registration: email + password + displayName
- Login: email + password
- Email must be unique, normalized, and verified
- Password reset via email
- Roles: `user`, `editor`

### B. Guanli AdminAccount (管理后台)
- Login: username + password
- NO email required
- Separate from frontend User
- Roles: `admin`, `reviewer`, `operator`

### Prohibitions
- Deleting User.email to satisfy guanli's no-email requirement
- Using email as username for guanli
- Using displayName for login
- Sharing JWT between frontend and backend
- Mixing the two account systems in one table

## 4. Role System Reconciliation

### Current (broken) state:
- User.role can be: `user`, `editor`, `admin`, `viewer`, or anything — no constraint
- Admin access determined by User.role === 'admin'
- No separation between frontend and backend roles

### Required state:
| System | Model | Allowed roles |
|--------|-------|--------------|
| Frontend | User | `user`, `editor` |
| Backend | AdminAccount | `admin`, `reviewer`, `operator` |

- User.role === 'admin' must NOT grant guanli access
- AdminAccount token must NOT work on frontend endpoints
- Remove `admin`, `viewer` from User.role allowed values
